REST Api

A REST Api is a web API that conforms to the REST architectural style. Thus
there are principles and conventions to be followed for an API to be
a RESTful api.

Guiding principles of REST

  • • Uniform Interface
  •   • Apply the principles of generality
  • • Client-Server design pattern
  •   • Enforces the seperation of concerns, helps client and server evolve independently.
  • • Stateless
  •   • Each request from client to server must contain all information neccesary
    to understand/complete the request.
  • • Cacheable
  •   • A response should implicitly or explicitly label itself as cacheable or non-cacheable
  • • Layered System
  •   • Architecture should be composed of hierarchical layers by constraining component behavior.
  • • Code on demand (optional)
  •   • Allows client functionality to extend by downloading and executing code in scripts/applets.

Naming conventions

REST resources has naming conventions, because it enforces the best design
decisions in the long term, thus helping with longetivity.

  • • Use nouns to represent resources
  •   • example: http://api.example.com/user-management/users/:id
  • • Client-Server design pattern
  •   • Enforces the seperation of concerns, helps client and server evolve independently.
  • • Document
  •   • Use 'singular' name to denote document resource archetype.
  • • Collection
  •   • Use the plural name to denote the collection resource archetype. eg. Users instead of user.
  • • Store
  •   • Use plural name to denote store resource archetype.   • http://api.example.com/user-management/users/:id/wishlists
  • • Controller
  •   • Use verb to denote controller archetype, eg. checkout, play etc..

Other than that, theres a couple of more stuff.
Use forward slash, but no forward slash at the end of the URI.
Use hyphens to improve readability.
No underscores.
Lowercase letters are preffered.
Never use CRUD functionality names in URIs.
